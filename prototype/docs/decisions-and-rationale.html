6. Prompt architecture 6 decisions

How prompts are composed per turn. Each pattern has its own prompt — different jobs need different tone, control flow, and success criteria. One mega-prompt couldn't serve all six.

#DecisionRationaleAlternatives consideredSource
PR1Layered prompt composition per turn — system + classifier + active-pattern + tool defs + retrieved contextSeparates cross-cutting concerns (brand, safety, identity) from pattern-specific behaviour. Caching applies to the stable layers; only the active-pattern + retrieved layers swap per turn.Single mega-prompt with all pattern logic embedded; per-topic promptsThis session · industry practice
PR2Classifier prompt returns typed JSON — {pattern, topic, purpose, risk}Structured output enables deterministic routing. The orchestrator's switch on pattern is a typed lookup, not a free-text parse.Free-text classification; embedding-similarity routing; rule-based intent matchingPA · this session
PR3Each pattern has its own dedicated prompt — in src/data/skills/<pattern>/prompts.mdInform-and-Act is "probe-then-route"; Express is "slow down, listen"; Decide is "eligibility → confirm → lodge". Different jobs, different tone, different success criteria. One prompt can't serve all six.Single prompt with pattern as a parameter; per-topic prompts; per-persona promptsThis session
PR4Cross-cutting concerns live in the system prompt — brand voice, safety rails, identity, distress overrideConsistent voice across patterns. Safety can't be opted-out by a pattern. Identity and tone are not duplicated per skill — less drift, easier audit.Duplicate in each pattern prompt (drift risk); compose programmatically (audit harder)Industry practice
PR5Anthropic Skills folder format — YAML frontmatter + markdown body per skillStakeholder-authorable (Student Experience can read/edit a prompts.md without TypeScript). Interoperable with ~/.claude/skills/ standard. Same format jcode and Claude Code adopt.Hard-coded TS constants; YAML-only; JSON configGH · anthropics/skills
PR6Prompts versioned in git; evaluated against the 11 scenarios as fixturesPrompts evolve. A change is acceptable iff scenario fixtures still pass deterministically.
