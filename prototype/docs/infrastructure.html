1. Topology — the big picture

Public traffic enters at Front Door. Identity is Entra. Compute is Container Apps + Static Web Apps. Application services (orchestrator, retrieval pipeline, classifier, safety subsystem) run inside compute. Three MCP tool servers (also Container Apps) sit between the orchestrator and external LTU systems, exposing tools via HTTP/SSE. Repositories span Azure-managed (Postgres, Redis, Blob), LTU systems (SIS via SAP PO), and external providers (LLM API).

8. MCP tool servers

Three MCP (Model Context Protocol) servers run as Container Apps services within the same environment as the orchestrator. They are internal only — no public endpoint. The orchestrator is the MCP client; tool calls from the six pattern prompts route through these servers before reaching external LTU systems. This makes the tool implementations independently deployable, separately scalable, and reusable across any future agent without modifying orchestrator code. CLI-binary tools (subprocess from the orchestrator) were considered and rejected for the reasons recorded in T19 — connection lifecycle against SAP PO and Postgres at census-load spikes, and the loss of a per-tool auth boundary independent of the orchestrator.

ServerShapeTools exposedDownstreamWhy
ltu-records-mcp OSS Azure records_lookup · check_eligibility SAP PO → SIS read path. Status-Check and Decide eligibility queries.
ltu-actions-mcp OSS Azure lodge_action (defer · withdraw · transfer · change-load · special-consid’n) SAP PO → SIS write path. Decide pattern Action API calls.
ltu-wellbeing-mcp OSS Azure request_advisor_handoff · log_distress_signal Contact-centre platform (advisor routing) + Redis hold state. Express pattern escalation path.
retrieve_knowledge is NOT a separate MCP server. The retrieval pipeline (Docling · LlamaIndex · BGE reranker · Hybrid BM25+vector) runs inside the orchestrator Container App. retrieve_knowledge is a direct in-process call to the pipeline — no HTTP hop, no extra service to manage.
Phase 2 deferral: ltu-outbound-mcp. Nudge and Check-in patterns use send_outbound_message (email · LMS · SMS). A fourth MCP server (ltu-outbound-mcp) will wrap this when proactive patterns ship. Not needed for the reactive Phase 1 baseline.

12. Resilience & DR

Single-region (Australia East) at launch. Geo-redundant Blob for durability. PITR on Postgres for 35 days.

Backup & recovery

  • Postgres — automated daily backups; point-in-time recovery to 35 days; geo-redundant backup storage.
  • Blob — geo-redundant storage; versioning enabled on Knowledge corpus container.
  • Redis — cache; no backup required. State that matters lives in Postgres.
  • Key Vault — soft-delete + purge protection; geo-redundant by default.
  • Audit log — append-only; daily snapshot to Blob Archive (SEC4).

14. Cost shape

Monthly cost for the Phase 1 production deployment at ~30k students. Big-ticket items: LLM API calls (capped by per-student budgets) and the primary database. Compute scales to zero during quiet periods. LLM costs are the most variable line — and the most controllable. The provider is configurable via models.config.ts (T15 · T16): Anthropic Claude, OpenAI GPT-4o, Google Gemini, GLM (Zhipu), Mistral, self-hosted Ollama/vLLM — any provider that speaks the LLMClient interface can be slotted in without touching orchestrator code. 15. Scale characteristics

Horizontal (compute)

  • Orchestrator — Container Apps autoscale on HTTP request rate. Scale-to-zero at low traffic; max ~10 replicas raisable.
  • Frontend — Static Web Apps CDN scales globally.
  • BGE reranker — sidecar can scale with orchestrator; or AzureML serverless absorbs spikes.

Vertical (data + cache)

  • Postgres — start 2 vCore, scale to 4–8 at semester start. Storage autogrows.
  • Redis — 6 GB Premium at launch; scale up without downtime.
  • Connections — PgBouncer in front of Postgres if connection count climbs.

Anticipated load shapes

  • Steady state — ~50–200 concurrent students; orchestrator at 1–2 replicas.
  • Census + deadline waves — 5–10× spike for 2–3 days; orchestrator scales to 5–10 replicas; database vCore up.
  • Express-pattern surges — handoff held states require advisor availability, not infrastructure scaling. Capacity managed by LTU's contact-centre platform / staff scheduling.
