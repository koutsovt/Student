Methodology · repeatable process for AI assistant projects

Conversational AI Engagement Playbook

Eight phases — from reading a brief to handing a stakeholder a PDF — distilled from a completed conversational-AI engagement. Each phase captures the thinking steps, the key outputs, and a heuristic you can carry into the next project. The worked examples throughout come from a higher-education chatbot replacement engagement and are presented generically.

Version 1.0 · May 2026 · domain-neutral · no client data

1 Intake & problem framing

The most important 30 minutes of the engagement. Read the brief to extract the current system's failure mode, not just its feature list.

What it is

Before designing anything, you need one clear sentence that describes what the incumbent system does wrong — not what it lacks. Features can be added; a wrong model can only be replaced.

Why it matters

Every subsequent decision flows from the failure-mode statement. The conversation patterns you choose, the retrieval strategy, the memory model — all of them exist to solve the one thing the current system cannot. If you skip this step, you end up building a better version of the wrong thing.

Worked example — higher-education chatbot

The incumbent chatbot was an IBM Watson-based FAQ-matcher sitting on the institution's student portal. Students typed questions; the system matched to one of ~20 sample questions per FAQ article and returned the article.

The failure mode wasn't "not enough FAQs" — it was purpose-blindness. A question like "what is the student fee?" could mean: (a) I want to understand what it is, (b) I need to pay it, or (c) I'm eligible to defer it and want to know how. The incumbent answered all three identically — with the definition article — because it classified by topic, not by intent.

This one sentence unlocked everything: "The surface query masks the real need — the system must classify intent before delivering content."

How to run it

  • Read the brief twice: first for context, second specifically hunting for failure-mode evidence (user complaints, escalation rates, unsatisfied queries, workarounds)
  • Ask: what does the current system do when the student/user's real need differs from their surface query?
  • Resist the urge to jump to solutions — stay in the problem space until the failure mode is crisp
  • Write a single paragraph. If it takes more than a paragraph, the failure mode isn't clear yet

Key outputs

  • One-paragraph failure-mode statement (the north star for the whole engagement)
  • List of the incumbent system's constraints that are worth preserving (integrations, brand voice, escalation paths)
Heuristic "The surface query masks the real need. Ask what's needed before answering."

2 Conversation pattern design

Once you have the failure mode, design the conversation model. Patterns classify intent, not topic — the same six handle every subject.

What it is

A conversation pattern is an interaction model — a reusable shape that defines what the assistant is trying to do in a turn, not what it's talking about. The six patterns below cover the full intent space for service-delivery assistants:

Inform-and-Act
Reactive
Probe purpose before delivering content. Surface queries hide different real needs — ask before answering.
Status-Check
Reactive
Look up this person's record and reply with their answer — not a generic article.
Decide
Reactive
Walk the user through a consequential action in chat. Eligibility → confirm → lodge.
Express
Reactive
Recognise distress or strong emotion, listen, surface support, hand off in-channel when needed. Never rush.
Nudge
Proactive
Reach the user before a deadline misses them. Outbound message in their channel, click-through into chat already in flight.
Check-in
Proactive
When a behavioural signal flags a user as potentially needing support, open warmly. No agenda. No pressure.

The proposing-mode axis

Each pattern has a distinct proposing mode — the stance the assistant takes when it initiates or shapes a response:

Inform-and-ActProbe — ask what they really need before delivering anything
Status-CheckCheck — retrieve and confirm the specific record state
DecideConfirm — present eligibility result and get explicit confirmation before lodging
ExpressOffer — surface soft options without directing; let the user choose
NudgeUrgency — contextual, specific, time-bound; not generic reminders
Check-inOpen — no agenda, warm, let the user define the direction

Phase sequencing

Reactive patterns (Inform-and-Act, Status-Check, Decide, Express) launch first — they replace the incumbent system. Proactive patterns (Nudge, Check-in) require persistent memory and a scheduler; they ship in a follow-up release once the reactive baseline is live and trusted.

Routing graph

Patterns hand off to each other via explicit transfer calls. The typical graph: Nudge → Decide; Check-in → Express or Inform-and-Act; Inform-and-Act → Status-Check or Decide; any pattern can escalate to Express. Mapping this graph early surfaces which handoff states need UI treatment (e.g. the Express held-state while an advisor joins).

Worked example — higher-education chatbot

The fee deferral question ("what is the student services fee?") was the canonical example. A purpose-blind system returned the definition article to all three variants of the question. With pattern classification:

  • Intent = understand → Inform-and-Act (hybrid retrieval, policy article)
  • Intent = check my status → Status-Check (records lookup, "you owe $X, due 14 May")
  • Intent = defer it → Decide (eligibility check, confirm, lodge deferral via Action API)

One surface query, three correct responses — none of them the incumbent's generic article.

Key outputs

  • Pattern map: which six patterns apply, with domain-specific examples for each
  • Routing graph showing handoff paths between patterns
  • Launch vs. follow-up split (reactive first, proactive second)
Heuristic "Classify intent, not topic. The same six patterns handle every subject area."

3 Persona & scenario design

Three personas × three lifecycle stages = the scenario matrix that drives the prototype and the workshop.

What it is

Personas are archetypes that represent the axes of variation in the user population — not average users, but users who stress-test the design in different ways. Scenarios are scripted interactions at the intersection of a persona and a lifecycle stage.

Why it matters

A prototype that only demonstrates one user type in one context gives stakeholders false confidence. The matrix forces the design to hold across different digital fluency levels, life circumstances, and points in the user journey. It also gives the workshop audience something to project onto — real archetypes prompt real conversation.

Persona design principles

  • Choose three archetypes that span the key axes: digital confidence, life stage, geographic/access context
  • Give each a name, a short biography, and a dominant tension (the thing that makes their relationship with the system hard)
  • Personas should be different enough to produce different routing paths — if two personas behave identically in every scenario, merge them
Worked example — higher-education chatbot

Three personas spanning the key axes of the student population:

  • International undergraduate — high digital confidence, unfamiliar with local institutional norms, financial stress around fee deadlines, uses the portal late at night across timezones
  • Mature online learner — moderate digital confidence, juggling work and family, disengages from Moodle under pressure, needs Check-in and proactive support more than reactive Q&A
  • Regional first-in-family student — lower institutional familiarity, on-campus intermittently, sanction/hold scenarios create disproportionate anxiety

Scenario matrix

Map each persona across three lifecycle stages (e.g. early term / mid-late term / end of term). Each cell in the 3×3 grid is a candidate scenario. Add two edge cases (crisis / extreme deadline pressure) to reach ~11 scenarios — enough to cover the pattern space without workshop fatigue.

Canonical worked example

Pick one scenario as the primary demo — the one that best demonstrates the failure-mode fix. In the higher-education engagement, this was the fee deferral workflow: a proactive Nudge outbound, click-through into Decide, eligibility confirmed, action lodged in chat. It demonstrated three patterns in sequence and made the value proposition concrete.

Key outputs

  • Three persona cards (name, biography, dominant tension, typical query patterns)
  • Scenario matrix (3×3 grid + 2 edge cases = ~11 scenarios)
  • One designated canonical worked example as the primary workshop demo
Heuristic "3 personas × 3 lifecycle stages = 9 cells. Add 2 edge cases = 11 scenarios. More is noise."

4 Technical framework research

Evaluate agent frameworks before committing. The wrong framework is harder to undo than the wrong model.

What it is

A structured survey of the open-source and commercial agent framework landscape, assessed against the specific requirements of the conversation model you've designed. Not a generic technology review — a fit-to-purpose evaluation.

The three evaluation axes

OSS maturity
Stars, contributors, release cadence, breaking-change history. A 5,000-file framework that moves fast is a maintenance burden, not a shortcut.
Pattern fit
Does the framework's mental model match yours? Frameworks built for single-agent tasks need re-shaping for multi-pattern routing. That re-shaping is hidden debt.
LLM abstraction quality
Can you swap the underlying model without touching orchestration code? Provider lock-in surfaces as a cost and compliance risk within 12 months.
Memory model
How does the framework handle per-user state, session continuity, and long-term memory? Bounded-by-design memory is safer than unbounded accumulation.
Worked example — higher-education chatbot

Four frameworks evaluated against the six-pattern model:

  • Hermes Agent — adopted for memory architecture (USER.md + MEMORY.md + SQLite FTS5 + Honcho external provider). Bounded-by-design memory, agent-curated facts, autonomous skill curator. ~16 source files — minimal surface area.
  • jcode patterns — adopted for two-tier safety (auto-allowed vs. requires-permission) and memory graph approach. Direct fit for the Express distress-hold requirement.
  • Anthropic Agent SDK (TS) — adopted as the runtime. First-party caching, streaming, typed tool calls. Session stores ship as production-ready examples.
  • Mastra — rejected. ~5,930 source files; a full platform with deployer, codemod, agent-builder. Adopting it means taking on a stack, not a library.
  • LangGraph (Python) — rejected. Team context-switch from TypeScript prototype; framework opinions don't match the six-pattern surface.

Key outputs

  • Shortlist of 2–4 frameworks with adoption rationale
  • Explicit rejection log for frameworks considered and not taken (prevents re-litigating in future sessions)
  • Decision entries in the log (see Phase 6)
Heuristic "Evaluate by what a framework doesn't do as much as what it does. Complexity you don't need today is debt."

5 Architecture design

Five principles, one scope boundary, and a clear stance on existing client infrastructure.

The five design principles

Purpose-first retrieval
Not everything is RAG. Match retrieval strategy to pattern: hybrid search for Inform-and-Act, structured API for Status-Check, tool-call for Decide. One retrieval approach for all patterns is the incumbent system's mistake repeated.
Bounded memory
USER profile (~500 tokens) + MEMORY facts (~800 tokens) + on-demand session retrieval. Context is predictable and cost is bounded. Unbounded memory accumulates garbage and unpredictable behaviour.
Per-user state, not per-user process
At scale (30k+ users), long-lived per-user agent processes don't scale. Shared stateless runtime + per-user state slice keyed by user ID. Same architecture, different cost model.
Provider-agnostic LLM
All model calls go through a typed LLMClient interface. Default to the best available model; swap without rewiring the orchestrator. LLM landscape changes faster than engagement cycles.
Phase-scoped delivery
Define a strict Phase 1 boundary — what must work to replace the incumbent. Everything else is Phase 2 or later. Document Phase 2 items as deferred, not designed. Designing for Phase 2 before Phase 1 is live is premature architecture.

Handling existing client infrastructure

Clients arrive with existing systems: middleware, identity platforms, cloud infrastructure, analytics stacks. The playbook rule: evaluate before adopting, document what you reject and why.

  • If it's a genuine fit — adopt it, note the decision, and ensure the adapter layer stays inside your deployment boundary (e.g. a SAP PO adapter module inside your orchestrator pod, not a dependency on the external system's reliability)
  • If it's not a fit or not cost-effective — reject it explicitly with rationale. Rejected items in the decisions log prevent the same conversation next quarter.
  • If it's a partial fit — adopt the pattern, not the product. Azure AI Search is a better fit than pgvector for some clients; Copilot Studio multi-agent is not a fit if the cost model is 3× the custom orchestrator.
Worked example — higher-education chatbot

The client had three existing infrastructure patterns to evaluate:

  • SAP PO middleware — adopted. Action API calls route via the existing SAP Process Orchestrator event bus before reaching SIS. The adapter runs inside our orchestrator pod; SAP PO's reliability is outside our blast radius but its routing contracts are respected.
  • Azure RAG pattern — partially adopted. Azure AI Search preferred over pgvector for the knowledge corpus. Azure OpenAI added as a co-equal LLM provider alongside Anthropic (residency requirement).
  • Copilot Studio multi-agent — explicitly rejected. Additional ~$2,400/mo at 30k students × 10 turns; removes LLMClient abstraction; adds Microsoft lock-in. Custom orchestrator is cheaper and more portable. Documented in the rejected decisions section.

Key outputs

  • Architecture diagram (topology SVG: compute zone, repositories zone, external zone, observability zone)
  • Component table: every component with ownership tag, choice, and rationale
  • Phase 1 scope boundary explicitly stated
  • Existing-client-infrastructure evaluation in the decisions log
Heuristic "Don't design for Phase 2. Time is irrelevant with AI — sequence by dependency, not calendar."

6 Decisions log

Every non-obvious decision logged in real time. The rationale decays faster than the code.

What it is

A structured HTML document that records every architectural, technical, UX, and security decision made during the engagement — including what was rejected and why. Not a retrospective; written as decisions are made.

Why it matters

Three failure modes the log prevents: (1) re-litigating decisions already made when a new stakeholder joins, (2) losing the "why" when a junior developer reads the code six months later, (3) governance review asking "did you consider X?" — the answer is already in the document.

Format

Five columns per row: ID (series prefix + number) · Decision (what was chosen) · Rationale (why) · Alternatives considered (what was evaluated but not taken) · Source (brief, user request, research session).

Category series

Group decisions by category. Recommended series from the higher-education engagement:

  • A — Architecture (the model itself)
  • R — Pattern → retrieval mapping
  • K — Knowledge scope
  • P — Prototype tech stack
  • T — Production tech stack
  • PR — Prompt architecture
  • UX — UX, branding, tone
  • S — Scenario design
  • B — Borrowed patterns (open-source research)
  • BO — Build order / prioritisation
  • SEC — Security & compliance
  • X — Explicitly rejected options

Scale calibration

A complete engagement at this scope produces roughly 80–100 decisions. Don't aim for a number — log every decision that you can imagine being re-questioned. Trivial choices (file naming, import order) don't need logging; anything involving a trade-off does.

Key outputs

  • Decisions log HTML (self-contained, printable, linkable by section)
  • Rejected-options section as a permanent record of the roads not taken
Heuristic "Log decisions in real time, not retrospectively. The rationale decays faster than the code."

7 Infrastructure documentation

A self-contained HTML document — topology diagram, component tables, and rationale — that a cloud engineer, a security reviewer, and a delivery manager can each read usefully.

What to include

Fourteen sections cover the full Phase 1 scope: Topology (SVG diagram) · Compute · Retrieval pipeline · Repositories · Networking & edge · Ingestion pipeline · Identity · External integrations · Wellbeing/escalation handoff · Observability · Resilience & DR · Security controls · Cost model · CI/CD.

SVG topology conventions

Four zones, each a dashed-border rectangle with a distinct colour:

  • Azure Compute (dark navy) — Container Apps, Static Web Apps, orchestrator sub-components
  • Repositories (rose/pink) — every data store mapped (Postgres, Redis, Blob, Key Vault)
  • External · Outbound (purple) — LLM provider, client SIS/records, escalation handoff
  • Observability · Operations (dark navy) — Application Insights, Log Analytics, CI/CD, ACR

Colour coding by ownership: Azure-native (blue) · OSS library (green) · External SaaS (purple) · Client system (rose). Every box carries: component name, key attributes, and a decision reference ID.

Phase 1 scoping rule

If a component isn't needed to replace the incumbent system, it's not in the document. No Phase 2 placeholders, no "future state" boxes — they add noise and imply commitment. The document covers exactly what ships.

Client infrastructure integration

Client-owned systems (middleware, identity platforms, SIS) appear in the External zone with a client ownership tag. Where a client system runs inside our deployment (e.g. an adapter module in the orchestrator pod), it appears as a sub-component of the relevant compute box — not as an external dependency.

Key outputs

  • Infrastructure HTML document (self-contained, no build step needed)
  • SVG topology diagram embedded in the document
  • Component tables with ownership, choice, and rationale columns
  • Cost model section with per-student monthly estimates at scale
Heuristic "Every component needs a zone, a colour, and a 'why'. If you can't state the why, question whether it belongs."

8 Stakeholder briefing

A 10–12 page PDF for senior leadership, generated from an editable HTML source. The HTML is the living document; the PDF is the snapshot.

Page structure

Twelve pages, three audience layers:

  • Pages 1–4 (leadership) — Cover · Problem today (what the incumbent does wrong) · Solution overview (the six patterns, one diagram) · Value proposition (why this, why now)
  • Pages 5–10 (technical team) — Memory architecture · System architecture diagram · Personas (three cards) · Scenario matrix · How to run the workshop demo · Retrieval and model strategy
  • Pages 11–12 (governance) — Decisions summary · Security and compliance posture · Next steps and Phase 1 scope

Presentation principles

  • Lead with the failure mode on page 2, before any proposed solution — the audience needs to agree the problem is real before they'll evaluate the fix
  • One diagram per page maximum; diagrams without a caption that explains the "so what" are decoration
  • The canonical worked example appears twice: as a scenario card (personas section) and as a step-by-step walk-through (how to demo section)
  • Phase 1 scope is explicit — what's in and what's deferred. "We will do X" is better than "we might eventually do Y"

PDF generation

Build the briefing as an HTML file with print-optimised CSS (@page margins, break-after: page on each page div). Generate the PDF via Chrome headless:

"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --print-to-pdf=output.pdf --no-pdf-header-footer file://path/to/briefing.html

Regenerate on every content change — the PDF is always derived from the HTML, never edited directly.

Iteration cadence

Expect 3–5 rounds of review. Keep a version suffix in the page footer (e.g. v1.2). Each round: update the HTML, regenerate the PDF, share the PDF. Never share the HTML directly unless the recipient needs to edit it — PDFs prevent inadvertent changes and preserve formatting across devices.

Key outputs

  • Stakeholder briefing HTML (editable source, ~12 pages when printed)
  • Stakeholder briefing PDF (generated from HTML, version-suffixed)
Heuristic "Build the HTML first, generate the PDF last. The HTML is the living document; the PDF is the snapshot."
